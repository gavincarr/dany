# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Repository layout

Two CLI binaries share one library:

- `dany.go` (repo root) — package `dany`, the shared DNS query engine. Exposes `RunQuery` (typed-ANY aggregation, returns `([]Answer, []error)`), `Render` (turn `[]Answer` into the canonical tab-separated text), and `RunNXQuery` (NXDOMAIN probing), plus the `Query`, `Resolvers`, `Answer`, `QueryError` types and the `DefaultRRTypes` / `SupportedRRTypes` / `DNSSECRRTypes` / `DNSSECBundle` / `NXTypes` / `SupportedUSDs` constants. Imported by the CLIs as `github.com/gavincarr/dany`.
- `render_json.go` (repo root) — defines the shared typed envelope used by every structured renderer. Exposes `BuildOutput` (assemble the typed `Output` envelope from `[]Answer` + `*Query` + `[]error`), `RenderJSON` (NDJSON-ready `\n`-terminated wrapper), the `Output` / `OutputAnswer` / `OutputError` envelope types, and one `*Data` struct per supported RR type. `SchemaVersion` is bumped only for breaking changes. Struct fields carry parallel `json:` and `yaml:` tags (snake_case, identical names) so JSON and YAML render to the same field names.
- `render_yaml.go` (repo root) — the YAML renderer. `RenderYAML` calls `BuildOutput` and marshals via `gopkg.in/yaml.v3`, prefixing each document with `---\n` so multi-hostname output is a valid multi-doc YAML stream.
- `render_unmarshal.go` (repo root) — the decode side. Marshalling an `OutputAnswer` works transparently because `encoding/json`/`yaml.v3` reflect the concrete `*Data` value in the `interface{}` field; *un*marshalling can't, and would otherwise flatten `Data` into a `map[string]interface{}`. Custom `OutputAnswer.UnmarshalJSON`/`UnmarshalYAML` fix this: they decode the envelope, then re-decode the raw `data` payload into the concrete `*Data` struct keyed off the answer's own `type` discriminator. `newData(typeName)` is the single source of truth mapping a type name to its `*Data` struct, shared by both decoders (and the marshal-side counterpart of `marshalData`). So the envelope is a faithful round-trip: a value that went out as `MXData` comes back as `MXData`, not a map. Unknown types fall back to a generic decode (forward-compat), not an error.
- `cmd/dany/main.go` — the `dany` CLI: simulates DNS `ANY` queries by firing the configured RR types (default `A,AAAA,HTTPS,MX,NS,SOA,TXT`) concurrently and aggregating results. `-f/--fmt {text,json,yaml,yml}` selects the renderer; structured modes (`json` / `yaml` / `yml`) fold errors into the envelope instead of going to stderr.
- `cmd/dnx/main.go` — the `dnx` CLI: takes hostnames (args or stdin) and reports those that return NXDOMAIN. For safety it runs all `NXTypes` (`MX,NS,SOA`) concurrently per hostname and only reports a host as NX if *every* type returns NXDOMAIN (`RunNXQuery` returns `len(NXTypes) - nxcount`).

Module path: `github.com/gavincarr/dany` (Go 1.25, deps: `miekg/dns`, `jessevdk/go-flags`, `gopkg.in/yaml.v3`).

Versioning: `internal/version/VERSION` is the single source of truth — a `//go:embed`'d plain-semver string read into both CLIs (`dany --version` / `dnx --version`). `scripts/release.sh <version>` syncs it with `debian/changelog`, commits, and tags. The changelog is the *structural* gate: the release script refuses to tag unless the head stanza version matches the requested arg, so there is no path to releasing a version that hasn't been written about.

## Build / test commands

```bash
# Build both binaries (run from each cmd/ dir)
cd cmd/dany && go build
cd cmd/dnx  && go build

# Run all (offline) tests
go test ./...

# Also run the live-DNS smoke tests (requires network + 8.8.8.8 reachable)
go test -tags integration ./...

# Run one test
go test -run TestRunQuery_Basic ./...
```

## Test layout

Default `go test ./...` is **fully offline**. The library's `RunQuery` and `RunNXQuery` are exercised against `internal/testdns`, an in-process `dns.Server` that returns canned RRs you `Add()` per (name, type). Real-world DNS drift can never break the default suite.

The two `cmd/*/integration_test.go` files are gated behind `//go:build integration` and hit real DNS (8.8.8.8) for a stable RFC 2606 reserved domain. They're a wire-format smoke check, not regression coverage — keep them small.

## Architecture notes worth knowing before editing `dany.go`

- **TCP by default.** `RunQuery` sets `client.Net = "tcp"` unless `Query.Udp` is true, because TXT/DNSKEY responses are often too big for UDP. Don't flip this default casually.
- **Data vs. rendering.** `RunQuery` returns raw `[]Answer` (each carrying queried Type, queried Hostname, and the raw `dns.RR`). Two renderers consume that: `Render(answers, tagHostname)` produces canonical tab-separated text (sort, tab layout, hostname tag prefix, PTR fold-in); `RenderJSON(answers, q, errs)` produces the typed JSON envelope via `BuildOutput`. The query path does no string formatting; renderers do no I/O. Adding a third format (yaml, etc.) means a new `RenderX` that calls `BuildOutput` and marshals — the typed envelope is format-agnostic. BuildOutput deduplicates its assembled answers on RRset identity (Type, Name, Class, Rdata), keeping the lowest TTL — the structured parallel to the text renderer's line-level `seen`-map dedup — so genuine wire-duplicate RRs collapse to one entry (RFC 2181 §5) while name-distinct records (www/apex, per-IP PTRs, CNAME hops) are preserved.
- **Errors are structured.** `RunQuery` / `RunNXQuery` errors are `*QueryError` (Type, Hostname, Code, Err). `Code` is a stable string — DNS rcode names (`NXDOMAIN`, `SERVFAIL`, ...) plus `EXCHANGE_ERROR` (transport-level) and `UNSUPPORTED_TYPE`. `Error()` preserves the historical `Error on <type> lookup for "<host>": <…>` format and `Unwrap()` keeps `errors.Is(err, ErrNXDomain/ErrServFail)` working. The JSON renderer unpacks `QueryError` into `OutputError{type, hostname, code, message}`; text consumers just print `err.Error()`.
- **Concurrency model.** `RunQuery` fans out one goroutine per RR type (plus one per USD probe if `q.Usd`) into a buffered `stream`; the main loop drains exactly `count` results or bails at a 10s wall-clock timeout (`timeoutSeconds`). Per-client dial/read/write timeout is `timeoutSeconds/2`. Same shape in `RunNXQuery` over `NXTypes`. **Error suppression is a per-`lookup` argument, not shared mutable state.** `lookup`/`dnsLookup` take an `ignoreErrors bool`: apex query types pass the caller's `q.IgnoreErrors` (snapshotted once before any goroutine launches), while best-effort USD/www probes pass `true` unconditionally so a missing `_dmarc`/`www` doesn't fail the query. Do **not** revive the old pattern of mutating `q.IgnoreErrors = true` before launching the probe goroutines — it raced with the apex goroutines' reads *and* leaked suppression onto the apex queries, silently swallowing their real SERVFAIL/REFUSED rcodes. Note `usd` and `ignoreErrors` are distinct args: www passes `usd=false, ignoreErrors=true` (best-effort but not an empty-record existence signal).
- **Adding a new RR type** requires five coordinated edits: add to `SupportedRRTypes`; text path — add a `formatX(rrtype, *dns.X) string` helper and a `case *dns.X:` in `formatAnswer`'s dispatch (`dany.go`); structured marshal path — add an `XData` struct with parallel snake_case `json:` and `yaml:` tags and a `case *dns.X:` in `marshalData` (`render_json.go`); structured unmarshal path — add a `case "X": return &XData{}` arm to `newData` (`render_unmarshal.go`) so the decoders can reconstruct the typed payload. No `lookup()` change needed — it uses `dns.StringToType` to map the type name to the wire-format constant. Then add a golden test for text plus a row in the `TestBuildOutput_RRTypeDataShapes` table and a case in `TestOutputAnswer_RoundTrip` (the latter is the drift guard that fails if `marshalData` and `newData` disagree). Together these cover both JSON and YAML — they share the typed envelope.
- **SVCB/HTTPS (RFC 9460).** `HTTPS` (type 65) is in the default set; `SVCB` (type 64, the generic form, usually only found under `_service._proto` prefixes) is `--all`-only. Both share one code path because `dns.HTTPS` embeds `dns.SVCB`: `formatSVCB(rrtype, *dns.SVCB)` and `svcbData(*dns.SVCB)` take the embedded struct, and `formatAnswer`/`marshalData` each carry *two* cases (`*dns.HTTPS` passes `&rr.SVCB`, `*dns.SVCB` passes `rr`) — the one deviation from the one-type-one-case recipe above. The single `SVCBData` struct serves both types. SvcParams flatten to an ordered `[]SVCBParam` (`key`/`value`) rather than a map; the record's wire order is canonical ascending-key order (RFC 9460 §2.2), so output is deterministic without an explicit sort. Values are each param's presentation form (`alpn` → `h2,h3`); the `ech` param's opaque base64 ECHConfig is surfaced verbatim.
- **CNAME handling.** `dnsLookup` chases the CNAME target for any non-CNAME request type, but *preserves* each traversed CNAME: the hop RR is prepended to the resolved answers and surfaces as an `Answer` tagged with the *queried* Type (e.g. `A`) while its RR is `*dns.CNAME`. That `Type != "CNAME"` + `*dns.CNAME` mismatch is the discriminator for "chain hop": `formatAnswer` folds these out of text (the resolved target records are already present), while the structured renderers emit them as their own `CNAME` `OutputAnswer` (owner name in `name`, target in `rdata`/`data.target`) so JSON/YAML capture the full mapping for archival. Explicit `-t CNAME` queries are *not* chased (`rrtype != "CNAME"` guard), so their answers carry Type `"CNAME"` and render normally. Multi-hop chains accumulate one CNAME answer per level.
- **PTR enrichment** (`-p/--ptr`) is fired inside the A/AAAA `lookup()` goroutine after the address records return, via `ptrLookupAll` → goroutine-per-IP `ptrLookupOne`. PTR responses travel back as their own `Answer`s (Type `"PTR"`, Hostname `=` the IP). Text `Render` folds them into the matching A/AAAA line — they are never emitted as standalone PTR rows. The JSON renderer keeps them standalone (one `OutputAnswer` per PTR) with `data.ip` carrying the source IP for downstream joining.
- **DNSSEC split (`--all` vs `--dnssec`).** DNSSEC records are triaged by the RR-selection rule (low-frequency, non-opaque, human/agent-useful). `DNSKEY` and `DS` — the chain-of-trust summary answering "is this zone signed, how, and is the delegation secured" — are in `SupportedRRTypes`, so `--all` shows them. The opaque, constantly-re-signed signing records `NSEC`/`RRSIG` are **not** in `SupportedRRTypes` (kept out of `--all`). Two DNSSEC constants split the remaining roles: `DNSSECRRTypes` (`DNSKEY,DS,NSEC,RRSIG`) is the full renderable set, merged into `parseOpts`'s type-validation map so an explicit `-t NSEC`/`-t RRSIG` stays valid; `DNSSECBundle` (`DNSKEY,DS,NSEC`) is what the `--dnssec` flag appends. `--dnssec` adds `DNSSECBundle` to whatever base set applies (default/`-t`/`--all`), deduped, in `parseOpts` *after* `WwwTypes` is fixed so www probes never inherit DNSSEC types. **RRSIG is deliberately excluded from `DNSSECBundle`:** a bare `QTYPE=RRSIG` query almost always SERVFAILs on a recursive resolver (an RRSIG signs another RRset and isn't independently validatable, so validating resolvers refuse the meta-query), so bundling it would add a near-guaranteed error line. It's still reachable via `-t RRSIG` (e.g. against an authoritative server, which can answer it).
- **USD empty-non-terminal.** `--usd` probes TXT at fixed underscore labels — the email auth/security set (`_dmarc`/`_domainkey`/`_mta-sts`/`_smtp._tls` TLSRPT/`default._bimi` BIMI) plus `_atproto` (Bluesky handle identity). Labels may be multi-level (`_smtp._tls`, `default._bimi`); the probe just prepends the label to the domain, so adding one is a single `SupportedUSDs` entry with no other code change. The `--help` USD line renders `SupportedUSDs` dynamically. A probe that returns NODATA (NOERROR + no records — a name that exists only because names below it do, e.g. `_domainkey.<domain>` when `<selector>._domainkey.<domain>` DKIM keys exist) is surfaced as a record-less `Answer{Empty: true}` (queried Type, `RR == nil`). NXDOMAIN stays omitted. Text renders it as `[present; no records]` (tag-aware: owner name in the value untagged, in the tag column under `--tag`); the structured renderers emit an `OutputAnswer` with an additive `present_empty: true` discriminator (no `SchemaVersion` bump). Only `--usd` probes can produce an `Empty` answer — ordinary NODATA on other queries is unchanged (renders nothing).
- **Resolvers** rotate round-robin via `Resolvers.Next()`. With multiple resolvers, `dany` re-picks per hostname; `dnx` re-picks per goroutine. `dnx`'s overall concurrency cap is `opts.Concurrency * resolvers.Length`. `Next()` is **concurrency-safe** — its `Index` read-modify-write is mutex-guarded, since `dnx` (and now `RunQuery` via `resolveServer`) call it from many goroutines on a shared `*Resolvers` (unguarded it both raced and could panic with an out-of-range index). The single-resolver fast path stays lock-free.
- **Server vs. Resolvers bridge.** The dial address is `Query.Server` (a `host:port` string) — `dnsLookup`/`nxlookup` dial it directly. `Query.Resolvers` is the higher-level round-robin set. `resolveServer(q)` in `dany.go` reconciles them: an explicit `Server` always wins (the CLIs set it per hostname, appending `dnsPort`), otherwise one resolver is picked via `Next()` and `dnsPort` appended. `RunQuery`/`RunNXQuery` each call `resolveServer` **once** per invocation (one resolver for all of a hostname's type lookups) and thread the result down, so a library consumer can set `Resolvers` alone — as the README example does — without also computing `Server`. Empty `Server` + no `Resolvers` yields a `""` dial target (per-type error), not a panic. `dnsPort` is a package `var` (default `"53"`) so in-process tests can point the bridge at testdns's random port.
- **Logging is all `slog`; don't reintroduce stdlib `log`.** Both CLIs' `setupLogger` calls `slog.SetDefault` — which also reroutes the stdlib `log` package's output through the slog handler at `Info` level. Since the default handler level is `Warn`, any `log.Fatal`/`log.Print` is silently dropped (the process still exits, so it looks like a no-message exit 1). Fatal paths therefore use `slog.Error(err.Error())` + `os.Exit(1)` (`Error` > `Warn`, so it's always shown). Non-error CLI output (usage text, per-hostname DNS errors in text mode) still writes directly to `os.Stderr`/`out` — that's intentional, not logging.

## CLI quirks (intentional, don't "fix")

`cmd/dany` still accepts two **deprecated** positional-arg forms and prints a deprecation warning at runtime:

- `@<ip>` — server, replaced by `-s/--server`
- bare `<RR>[,<RR>,...]` — types list, replaced by `-t/--types`

The detection logic in `parseArgs` (regex on `@` prefix, `.`, and `,`, plus a typeMap lookup) is what enables this. The `testMode` bool on `parseOpts`/`parseArgs` exists solely to suppress the warning text during tests — keep that plumbing if you refactor.
